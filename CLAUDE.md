# CLAUDE.md

## О проекте

kaif-radio — веб-плеер интернет-радио. Станции берутся из публичного API
radio-browser.info, пользовательские данные (аккаунты, избранное) — в Supabase.
MVP-жанр: lofi.

## Глобальная цель проекта

Построить надежную и быструю инфраструктуру, отказоустойчивую для пользователя, основанную
на современной стеке (Supabase Cloud, Redis, React/TypeScript и т.д; всё работает в docker изолированными частями).

Уже сделано:

- Backend живёт на отдельном арендованном сервере (VPS, Германия) и общается
  с radio-browser вместо фронта: ежечасовая выгрузка каталога в память,
  Redis-персистентность, перебор зеркал. Фронт ходит только в наш backend.
- Каждый сервис — в своём docker-контейнере (frontend / backend / redis).

Актуальные задачи:

- Реализовать MVP frontend части проекта
- Довести деплой: сборка образов в CI, домен + HTTPS (сейчас голый IP)

## Структура монорепозитория

- `frontend/` — SPA: React 19 + TypeScript + Vite, react-router 7, CSS Modules
- `backend/` — Fastify + TypeScript: каталог станций в памяти, Redis-персистентность
- `docker-compose.yml` — локальная разработка, `docker-compose.prod.yml` — сервер

Supabase — облачная (Supabase Cloud), локальной инсталляции нет.
Схема БД и RLS-политики пока живут только в веб-панели и в репозитории
не версионируются — долг на будущее (`supabase link` + `supabase db pull`).

## Архитектура данных: две дорожки

Данные ходят не по одной линии `frontend → backend → supabase`, а по двум
независимым дорожкам, разделённым по ответственности:

- **Дорожка A — пользовательские данные** (auth, favorites, профиль):
  `frontend ↔ Supabase` напрямую. Защита — RLS-политики, ключ `anon`.
  Auth-флоу — на gotrue, свой backend его НЕ дублирует.
- **Дорожка B — данные станций** (radio-browser, кеш, фильтрация):
  `frontend → backend → radio-browser / Redis`, при нужде backend пишет
  в Supabase под ключом `service_role`.

Границы безопасности:

- `service_role`-ключ живёт только в backend, во frontend не попадает никогда.
- Backend не строит свою авторизацию — он проверяет подпись JWT, выданного
  Supabase, и достаёт из него `user_id`.
- Фронт про radio-browser не знает вообще: единственное место, знающее адрес
  нашего backend — `entities/station/api.ts` (`VITE_API_URL`).

## Архитектура: FSD

Слои: `app / pages / widgets / features / entities / shared`.
Подход не жёсткий, но направление импортов соблюдается: сверху вниз,
слой не импортирует из слоёв выше себя.

Внутри слайса: `api / model / ui`, бизнес-логика — в хуках (`model/hooks`),
UI-компоненты логики не содержат.

## Правила кода

- **Только `type`, `interface` запрещён.**
- Определения типов всегда выносятся в отдельный файл types.ts, чтобы не засорять код
- Стили — только CSS Modules: `import styles from './X.module.css'`.
- SOLID, декларативность.
- YAGNI подход

## Декомпозиция (ВАЖНО)

Декомпозиция — не пожелание, а обязательное требование. Код обязан читаться.

- **Один модуль/компонент/функция — одна ответственность.** Если файл смешивает
  разные заботы (состояние домена + механика + обработка событий + сеть) — это
  сигнал разбить его, а не дописывать дальше.
- **UI-компоненты и провайдеры НЕ содержат механику.** Сложная логика (движки
  воспроизведения, обработка ошибок, таймеры, ретраи, сетевые процессы) выносится
  в отдельные хуки/модули в `model/`. Провайдер только собирает готовые части и
  раздаёт их через контекст.
- **Порог на глаз:** если внутри одного файла больше одного «почему это тут»,
  или его нельзя окинуть взглядом и сразу понять — декомпозируй.
- **Задачи декомпозируются так же, как код.** Крупную работу разбивать на
  маленькие шаги и делать по одному, а не вываливать всё одним куском.
- Предпочитать много маленьких понятных файлов одному большому «на всё».

## Отступы между блоками кода

Внутри функции код разделяется пустыми строками на логические шаги —
сплошные «кирпичи» текста запрещены:

- После guard-условия (early return) — всегда пустая строка.
- Между отдельными шагами функции (подготовка → действие → результат) —
  пустая строка; разные действия не слипаются в один блок.
- Короткие колбэки на 2–3 строки — единое целое, внутри них пустые строки
  не нужны.
- Эталон ритма:

```ts
const clear = () => {
	if (idRef.current === null) return;

	clearTimeout(idRef.current);

	idRef.current = null;
};
```

## Внешние API

- radio-browser.info: только на backend, в `radioBrowser/`. Зеркала перебираются
  по списку из `radioBrowser/data.ts` (de1 → nl1 → at1) — не хардкодить хосты
  по коду. Фронт в radio-browser не ходит.
- Supabase: клиент — единственный, в `shared/api/supabaseClient.ts`.
  Ключи — через `.env` (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`).
