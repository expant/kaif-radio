# CLAUDE.md

## О проекте

kaif-radio — веб-плеер интернет-радио. Станции берутся из публичного API
radio-browser.info, пользовательские данные (аккаунты, избранное) — в Supabase.
MVP-жанр: lofi.

## Глобальная цель проекта

Построить надежную и быструю инфраструктуру, отказоустойчивую для пользователя, основанную
на современной стеке (Локальный supabase, Redis, React/TypeScript и т.д; всё работает в docker изолированными частями).

Актуальные задачи:

- Реализовать MVP frontend части проекта

Заготовки идей отложенные на будущее:

- Клиент-браузер взаимодействует только через наш backend, который будет жить на отдельном арендованном сервере (например в Москве).
  Ответственность backend-а - взаимодействовать с api станциями, осуществлять оптимизацию передачи данных фронтенду (Redis, ежечасовая выгрузка актуальных станций, фильтрация работающих станций и тд).
- По аналогии с supabase, все остальные сервисы/фичи будут изолировано работать в своих собственных docker контейнерах

## Архитектура данных: две дорожки

Данные ходят не по одной линии `frontend → backend → supabase`, а по двум
независимым дорожкам, разделённым по ответственности:

- **Дорожка A — пользовательские данные** (auth, favorites, профиль):
  `frontend ↔ Supabase` напрямую. Защита — RLS-политики, ключ `anon`.
  Auth-флоу — на gotrue, свой backend его НЕ дублирует.
- **Дорожка B — данные станций** (radio-browser, кеш, фильтрация):
  `frontend → backend → radio-browser / Redis`, при нужде backend пишет
  в Supabase под ключом `service_role`.

Границы безопасности:

- `service_role`-ключ живёт только в backend, во frontend не попадает никогда.
- Backend не строит свою авторизацию — он проверяет подпись JWT, выданного
  Supabase, и достаёт из него `user_id`.
- `entities/station/resolveServer.ts` — временный протез: пока backend не готов,
  frontend ходит в radio-browser сам. С появлением backend эта логика уезжает
  на сервер, усложнять её на фронте не нужно.

## Архитектура: FSD

Слои: `app / pages / widgets / features / entities / shared`.
Подход не жёсткий, но направление импортов соблюдается: сверху вниз,
слой не импортирует из слоёв выше себя.

Внутри слайса: `api / model / ui`, бизнес-логика — в хуках (`model/hooks`),
UI-компоненты логики не содержат.

## Правила кода

- **Только `type`, `interface` запрещён.**
- Определения типов всегда выносятся в отдельный файл types.ts, чтобы не засорять код
- Стили — только CSS Modules: `import styles from './X.module.css'`.
- SOLID, декларативность, декомпозиция.
- YAGNI подход

## Внешние API

- radio-browser.info: сервер выбирается динамически через
  `entities/station/resolveServer.ts` (с фоллбеками и кешем).
  Не хардкодить конкретные хосты вида `de1.api...` в коде.
- Supabase: клиент — единственный, в `shared/api/supabaseClient.ts`.
  Ключи — через `.env` (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`).
